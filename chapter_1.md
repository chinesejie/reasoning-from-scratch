欢迎来到大语言模型的下一个阶段：**推理**。LLM 已经彻底改变了我们处理和生成文本的方式，但它们的成功很大程度上是由统计模式识别驱动的。推理方法的新进展使 LLM 能够应对更复杂的任务，例如解决逻辑谜题和多步数学问题。此外，推理是让 AI 智能体（agent）变得实用的关键技术，例如当智能体必须将任务分解为多个步骤、使用工具并从错误中恢复时。推理型 LLM 已经被应用于 OpenClaw 等智能体应用中。

在处理推理模型时，至关重要的是要知道如何自动检查答案在数学等任务中是否正确，以及如何将一个通用模型转化为更小的推理模型。

全书我们将使用数学示例，因为它们易于自动检查，但跟随本书学习并不需要高深的数学知识。理解推理方法才是本书的核心焦点。

本书侧重于实际应用和解释，面向 LLM 工程师、机器学习研究人员、应用科学家以及软件开发者。理解推理模型的构建方式有助于我们判断何时标准 LLM 已经足够，何时额外的推理机制值得增加复杂性、成本和延迟。它还有助于我们设计更好的评估方法、调试多步任务中的失败，并将推理技术适配到实际产品和研究工作流中。

## 1.1 在 LLM 语境下定义推理

在本书中，我使用的术语"推理"是在实用的工程意义上，而非哲学意义上。在 LLM 的语境中，**推理**指的是模型在产生最终答案之前生成中间步骤。

这些中间步骤可能以可见的逐步解释形式出现，也可能被包裹在特殊的标签中（如 `<think>...</think>`）并对用户隐藏。无论哪种情况，核心思想都是一样的：即鼓励或训练模型将 token 花在中间问题解决上，而不是直接跳到结论。

相应地，我使用**推理模型**来指一个通过训练或提示技术得到改进的 LLM，使其能够产生这样的中间步骤，这通常会提高其在编码、逻辑谜题和数学问题等复杂任务上的准确性。

在进入本书后续章节的代码部分之前，我将简要讨论改进这些推理行为的主要技术，以及它们与模式匹配和逻辑推理的关系。这将为进一步讨论 LLM 当前的构建方式、它们如何处理推理任务、以及它们擅长和不擅长什么奠定基础。²

> **思维链（Chain-of-Thought, CoT）**
> 
> 这种生成中间步骤的风格通常被称为**思维链（CoT）**。研究人员和工程师常说模型一步步地"思考"问题，意思是它使其中间推理过程变得明确且更易于理解。在本书中，我在这种常见的工程意义上使用"推理"和"思考"这两个术语。这并不意味着 LLM 实际上以与人类相同的方式进行推理或思考。

图 1.1 展示了传统 LLM 如何生成对用户问题的回答。

**图 1.1** 传统非推理型 LLM 如何用简短答案回应问题的简化示意图。

如图 1.1 所示，传统 LLM 可能不会展示它是如何得出答案的。虽然答案可能是正确的，但它无助于用户理解答案背后的过程。

图 1.2 展示了 LLM 中多步（CoT）推理的一个简单示例。³

**图 1.2** 推理型 LLM 如何使用思维链处理多步推理任务的简化示意图。模型不仅仅是回忆一个事实，而是结合几个中间推理步骤来得出正确结论。中间推理步骤可能展示给用户，也可能不展示，取决于具体实现。

如图 1.2 所示，LLM 生成的中间推理步骤看起来非常像一个人在大声表达内心想法。然而，这些方法（以及由此产生的推理过程）在多大程度上反映了人类推理，仍然是一个开放的研究问题，本书不试图回答这个问题。

虽然图 1.2 是思维链推理的典型示例，但重要的是要强调 LLM 推理不同于传统的确定性推理。

例如，符号逻辑引擎或定理证明器遵循严格的、基于规则的步骤，以保证一致性和正确性。符号逻辑引擎就像按照食谱做菜，每一步都是固定的，每次都必须产生相同的结果。用一个烹饪类比来说，如果你严格按照步骤执行，你总是能得到同一道菜。

与符号逻辑引擎相比，LLM **自回归地（autoregressively）**生成推理，这意味着它基于训练数据中的统计模式一次预测一个 token。因此，LLM 的"推理步骤"即使在看起来很有说服力的情况下，也不能保证在逻辑上是正确的。⁴

本书专注于解释和实现改进基于 LLM 的推理的基础技术，从而使 LLM 更擅长处理复杂任务。我希望通过亲自动手实践这些方法，你能更好地准备理解和改进那些正在开发中的推理方法，甚至探索它们与人类推理的比较。

> **LLM 推理 vs 人类推理**
> 
> LLM 中的推理过程表面上可能类似于人类思维，特别是在中间步骤的表达方式上。认识到一个关键区别很重要：人类可以通过有意识地应用逻辑规则或基于对世界的心智模型进行推理来从事**确定性推理**。确定性在这里意味着，如果我们从相同的事实出发并遵循相同的步骤，我们总是能得出相同的结论。相比之下，当前的 LLM 推理是**概率性的**，这意味着它基于训练数据中的统计模式一次生成一个 token，而不保证逻辑一致性。
> 
> 人类通常通过有意识地操纵概念、直观地理解抽象关系或从少量例子中泛化来进行推理。相比之下，LLM 通过从海量文本中拾取模式来工作，而不是依赖内置的推理规则或任何类型的有意识思考。
> 
> 简而言之，尽管增强推理的 LLM 的输出可以看起来像人类，但其底层机制有本质的不同，并且仍然是一个活跃的探索领域。

## 1.2 理解标准 LLM 训练流程

现在，让我们简要总结传统（非推理型）LLM 通常是如何训练的，这样我们就能理解它们的局限性在哪里。这个背景也有助于为后续关于模式匹配和逻辑推理之间差异的讨论奠定基础。

在应用任何特定的推理方法之前，传统 LLM 训练通常分为两个阶段：**预训练**和**后训练**，如图 1.3 所示。

一些近期论文也会在两者之间区分出一个**中训练（mid-training）**阶段，例如继续在代码、数学或长上下文数据上训练模型，但为了在本书中保持术语简单，我将这个阶段归入预训练。⁵

**图 1.3** 典型 LLM 训练流程概览。该过程从一个以随机权重初始化的初始模型开始，然后在大规模文本数据上进行预训练，通过预测下一个 token 来学习语言模式。后训练则通过指令微调和偏好微调来优化模型，使 LLM 能更好地遵循人类指令并与人类偏好对齐。

在典型 LLM 训练流程的预训练阶段，如图 1.3 所示，LLM 在海量无标注文本上进行训练，通常是数 TB 的数据或数万亿 token，包括书籍、网站、研究文章和许多其他来源。LLM 的预训练目标（目的）是学习预测这些文本中的下一个词（即 token）。

> **词与 Token**
> 
> **Token** 是语言模型处理的一小段文本单元。一个 token 可以是一个完整的词、一个词的一部分，甚至是一个标点符号，具体取决于所谓的**分词器（tokenizer）**如何将文本拆分。
> 
> 例如，句子 "An LLM can be useful." 可能被一个常见的分词器拆分为 "An"、" L"、"LM"、" can"、" be"、" useful" 和 "." 等 token。这些 token 随后被转换为模型可以摄入的数值 ID。
> 
> 分词器不是 LLM 本身的直接组成部分，但它仍然是 LLM 文本处理和生成流程中的关键组件。我们将在下一章看到分词器在实践中是如何工作的。⁶

当在 massive 数据集上进行预训练时，LLM 变得非常强大，这通常涉及数 TB 的文本（相当于数万亿 token）。这种训练需要数千个 GPU 运行数月，成本可能高达数百万美元。在这里，"强大"意味着 LLM 开始生成非常接近人类写作的文本。此外，在某种程度上，预训练后的 LLM 将开始表现出所谓的**涌现特性（emergent properties）**，这意味着它们能够执行一些没有被明确训练过的任务，包括翻译、代码生成等。

这些预训练模型仅仅作为后训练阶段的**基础模型（base models）**，后训练使用两种关键技术：**监督微调**（文献中常缩写为 SFT，也称为**指令微调（instruction tuning）**）和**偏好微调**（通常通过一种称为**人类反馈强化学习（Reinforcement Learning with Human Feedback）**的技术实现）来教 LLM 回应用户查询，如图 1.4 所示。

**图 1.4** 语言模型在不同训练阶段的示例回答。提示要求总结睡眠与健康之间的关系。预训练 LLM 产生一个相关但不够聚焦、没有直接遵循指令的答案。指令微调后的 LLM 生成一个简洁准确的摘要，与提示对齐。偏好微调后的 LLM 进一步改进回答，使用友好的语调和引人入胜的语言，使答案更具亲和力且以用户为中心。⁷

如图 1.4 所示，指令微调提升了 LLM 在个人助理类任务上的能力，例如问答、文本摘要和翻译等。偏好微调阶段则进一步优化这些能力。顾名思义，偏好微调帮助将回答调整为符合用户偏好。一些读者可能熟悉**人类反馈强化学习（RLHF）**等术语，这是实现偏好微调的具体技术。

简而言之，我们可以将预训练视为"原始语言预测"（通过下一个 token 预测），它赋予 LLM 一些基本特性和生成连贯文本的能力。后训练阶段则通过指令微调提升 LLM 的任务理解能力，并通过偏好微调优化 LLM 以创建具有更受青睐的风格选择的答案。

值得注意的是，即使是经过指令微调的模型也还不是"聊天机器人"。聊天界面增加了另一层，以交互式、多轮对话的方式引导模型的回答。这通常涉及系统提示、对话历史管理和其他编排（附录 G 中实现了其中一个示例）。

> **注**：上述预训练和后训练阶段在我的《从零开始构建大语言模型》（Manning 出版，http://mng.bz/orYv）一书中有详细介绍。你现在阅读的这本书不需要对这些阶段有详细的了解。我们将加载一个已经完成了上述昂贵的预训练和后训练阶段的模型，以便我们可以在后续章节中专注于推理模型特有的方法论。

## 1.3 用训练和推理技术改进 LLM 推理能力

LLM 语境下的推理在公众眼中随着 OpenAI 于 2024 年 9 月 12 日在 ChatGPT 中发布 o1 而变得流行，它普及了 LLM 中推理的概念。在发布文章（https://openai.com/index/introducing-openai-o1-preview/）中，OpenAI 提到："我们开发了一系列新的人工智能模型，旨在花更多时间思考后再作出回应。"

此外，OpenAI 写道："这些增强的推理能力对于你在科学、编码、数学和类似领域处理复杂问题时可能特别有用。"

几个月后，2025 年 1 月，DeepSeek 发布了 DeepSeek-R1 模型和技术报告（https://arxiv.org/abs/2501.12948），详细介绍了开发推理模型的训练方法论，这引起了巨大轰动，因为他们不仅免费且公开地提供了一个与专有 o1 模型竞争甚至超越其性能的模型，还分享了如何训练这种模型的蓝图。

我们的目标是通过从头实现类似的方法来理解这些用于开发推理模型的方法论是如何工作的。

开发和改进 LLM 推理能力的不同方法可以分为三大类，如图 1.5 所示。⁸

**图 1.5** 常用于提升 LLM 推理能力的三种方法。这些方法（推理计算扩展、强化学习和蒸馏）通常在常规训练阶段（初始模型训练、预训练以及指令和偏好微调的后训练）之后应用，但推理技术也可以应用于预训练后的基础模型。

如图 1.5 所示，这些方法通常应用于已经经历了常规预训练和后训练阶段（包括指令和偏好微调）的 LLM。以下列表简要介绍了这三种常见方法，我们将在本书后续章节中更详细地探讨。

1. **推理时计算扩展（Inference-time compute scaling）**（也常称为推理计算扩展、测试时扩展或其他变体）包括在**推理时**（当用户向模型提问时）改进模型推理能力的方法，而无需训练或修改底层模型权重。核心思想是用增加计算资源来换取改进的性能，这有助于通过思维链推理和各种采样程序等技术，使即使是固定模型也能变得更强大。这个主题将是第 4 章和第 5 章的重点。

2. **强化学习（Reinforcement Learning, RL）**指的是通过鼓励模型采取能获得高奖励信号的行动来改进其推理能力的训练方法。这些奖励可以很宽泛，例如任务成功或启发式分数，也可以定义得很窄且可验证，例如数学问题或编码任务中的正确答案。⁹

   与推理时计算扩展（可以在不修改模型的情况下改进推理性能）不同，RL 在训练期间更新模型的权重。这使模型能够通过试错来学习和优化推理策略，基于从环境中获得的反馈。我们将在第 6 章和第 7 章更详细地探讨 RL。

3. **蒸馏（Distillation）**涉及将强大的大模型学到的复杂推理模式转移到更小或更高效的模型中。在 LLM 的语境下，这通常意味着使用由更大、能力更强的模型生成的高质量标注指令数据集来进行监督微调（SFT）。这种技术在 LLM 文献中通常被称为**知识蒸馏**或简称为**蒸馏**。值得注意的是，这与深度学习中的传统知识蒸馏略有不同，在传统知识蒸馏中，较小的"学生"模型通常同时学习较大"教师"模型的输出和 logits。这个主题将在第 8 章中进一步讨论。

> **用于推理的强化学习与偏好微调**
> 
> 在开发推理模型的语境下，区分这里的 RL 方法与**人类反馈强化学习（RLHF）**很重要，RLHF 用于在开发传统 LLM 时的偏好微调阶段，如前面图 1.4 所示。
> 
> 两种设置使用相同的基础过程（RL），但它们主要在奖励的获取和验证方式上有所不同（RLHF 使用人类判断，而推理 RL 使用自动验证器或环境）。
> 
> RLHF 将明确的人类评估或模型输出的排名作为奖励信号，直接引导模型朝向人类偏好的行为。相比之下，推理模型语境下的 RL 通常依赖自动化或基于环境的奖励信号，这些信号可能更客观，但可能与人类偏好的一致性较低。例如，推理模型开发流程中的 RL 可能通过为正确性提供明确的奖励来训练模型在数学证明方面表现出色。相比之下，RLHF 会涉及人类评估者对各种回答进行排名，以鼓励与人类标准和主观偏好紧密对齐的输出。

## 1.4 模式匹配与逻辑推理

如上一节所述，在预训练期间，LLM 接触海量文本，并通过识别和重现数据中的统计关联来学习预测下一个 token。这个过程使它们能够生成流畅且连贯的文本，但它从根本上基于学习到的统计规律性，而非显式规则或保证的逻辑推断。

LLM 通过生成与训练期间看到的模式统计一致的文本延续来回应提示。这包括许多从前提推导结论的模式，看起来像是逻辑推断。关键区别在于，模型并不显式地表示前提并应用形式规则，而是从其训练数据中类似例子里预测可能的延续。

考虑以下示例：

**提示**
> The capital of Germany is…

**回答**
> Berlin.

一个 LLM 产生"Berlin"这个答案并不是在逻辑上推导出答案。相反，它是在回忆从训练数据中学到的强统计关联。这种行为反映了我们所说的**模式匹配**，即模型基于学习到的相关性来完成文本，而不是通过应用结构化的推理步骤。

但是，对于那些超越模式识别的任务呢？即正确答案依赖于从给定事实中得出结论的任务？这引向了另一种不同的能力：**逻辑推理**。

逻辑推理涉及使用规则系统地得出结论。与模式匹配不同，它依赖于中间推理步骤以及识别矛盾或基于形式关系进行推导的能力。

考虑以下提示作为示例：

**提示**: "All birds can fly. A penguin is a bird. Can a penguin fly?"

有两种方式来评估这个问题。¹¹

首先，在**封闭世界（仅提示）**设定下，从提示中的两个前提（主张、假设）（"All birds can fly" 和 "A penguin is a bird"），有效的答案是"Yes, a penguin can fly."

第二，在**开放世界（有背景知识）**设定下，如果我们还允许提示中未包含的背景知识（例如，企鹅不会飞），这个外部事实与从前提推导出的结论相冲突，如图 1.6 所示。一个推理系统应该注意到这种不一致，并要求澄清或弱化第一个陈述（例如，"Most birds can fly, with exceptions such as penguins"）。

**图 1.6** 矛盾的前提导致逻辑不一致。从"All birds can fly"和"A penguin is a bird"，我们推断出"Penguin can fly"。这个结论与既定事实"Penguin cannot fly"相冲突，从而产生矛盾。

图 1.6 展示了基于逻辑推理的系统如何处理前面介绍的"All birds can fly..."提示。

相比之下，统计（模式匹配）LLM 并不显式地追踪矛盾（如图 1.6 所示），而是基于学习到的文本分布进行预测。例如，如果"All birds can fly"这类信息在训练数据中被强烈强化，模型可能会自信地回答："Yes, penguins can fly."

在下一节中，我们将看一个关于 LLM 如何处理这个"All birds can fly..."提示的具体示例。¹²

> **逻辑推理与基于规则的系统**
> 
> 为什么显式基于规则的系统没有更受欢迎？基于规则的系统在 80 年代和 90 年代被广泛用于医疗诊断、法律决策和工程领域。它们仍然用于关键领域（医学、法律、航空航天），这些领域通常需要显式推断和透明的决策过程。它们很难实现，因为它们在很大程度上依赖人类精心设计的启发式规则。（启发式规则是简单的决策规则，能快速给出足够好的答案，但不保证最优。）相比之下，包括 LLM 在内的深度神经网络不实现手写规则；它们从数据中学习决策模式，并在规模化训练时可以高度灵活。

## 1.5 在没有显式规则的情况下模拟推理

我们看到了矛盾的前提如何导致逻辑不一致。传统 LLM 并不显式地追踪矛盾，而是基于学习到的文本分布生成回答。

让我们看一个具体的示例，如图 1.7 所示，展示一个非推理增强型 LLM（如 OpenAI ChatGPT 中的 GPT-4o）如何回应上一节讨论的"All birds can fly..."提示。¹³

**图 1.7** 语言模型（ChatGPT 中的 GPT-4o）如何对矛盾前提进行"推理"的示例说明。

图 1.7 中的示例显示，GPT-4o 似乎回答正确，尽管这个模型不像 OpenAI 的其他产品（如 o1、o3、o4-mini 和更新的 GPT-5）那样被认为是推理模型，那些产品是使用推理方法论明确开发的。

那么，4o 模型是如何生成它的答案的呢？这是否意味着 GPT-4o 在显式地进行逻辑推理？不，不一定。至少 4o 在熟悉的语境中非常有效地**模拟**逻辑推理。

GPT-4o 并没有实现显式的矛盾检查，而是基于概率加权的模式生成答案。如果训练数据包含大量纠正这个矛盾的实例（例如，像"penguins cannot fly"这样的文本），这种方法就足够有效，使模型学会"企鹅"和"不会飞"之间的统计关联。正如我们在图 1.7 中看到的，这使得模型能够在没有显式实现基于规则或显式逻辑推理方法论的情况下正确回答。

换句话说，模型隐式地识别出矛盾，因为它在训练期间频繁遇到这个确切的推理场景。这种有效性严重依赖于从大量接触训练数据中的类推理模式而建立的统计关联。¹⁴

因此，即使当传统 LLM 似乎在进行逻辑演绎（如图 1.7 所示），它也不是在执行显式的、基于规则的逻辑，而是在利用其海量训练数据中的模式。

尽管如此，GPT-4o 在此处的成功有力地说明了当在大规模上训练时，隐式模式匹配可以变得多么强大。这些基于模式的推理模型通常在以下场景中遇到困难：

- 逻辑场景是新颖的（在训练数据中从未遇到过）。
- 推理复杂度很高，涉及复杂的、多步逻辑关系。

> **逻辑推理与当前推理型 LLM 产品**
> 
> 虽然 GPT-4o 没有被官方标注为推理模型，但 OpenAI 提供了几个专门的推理模型，包括 o1、o3、o4-mini 和 GPT-5。此外，其他公司也一直在开发具有显式推理能力的 LLM。截至本书写作时，流行的例子包括 Anthropic 的 Claude 4、xAI 的 Grok 4、Google 的 Gemini 2.5、DeepSeek 的 R1、阿里巴巴的 Qwen3 等等。这些模型采用的技术是本书的重点。正如我们将看到的，这是通过实现基于规则的推理流程来实现的（图 1.6 展示了基于规则推理的一般概念）。相反，LLM 学习或改进其推理能力是通过改进的推理和训练方法论的结果。
> 
> 我们可以说，LLM 通过学习的模式来模拟逻辑推理，我们可以通过包括推理计算扩展和诸如强化学习等后训练策略在内的特定推理方法进一步改进它，但它们并没有在内部显式执行任何基于规则的逻辑。
> 
> 此外，值得提及的是，LLM 中的推理存在于一个频谱上。这意味着即使在 OpenAI 的 o1 和 DeepSeek-R1 等专用推理模型出现之前，LLM 就已经能够模拟推理行为了。例如，这些模型表现出与我们前面定义一致的行为，例如生成中间步骤来得出正确结论。我们现在明确标注为"推理模型"的，本质上就是这种能力的更精细版本。而这些改进的推理能力是通过利用特定的推理计算扩展技术（第 4 章和第 5 章）和有针对性的后训练方法（如强化学习，第 6 章和第 7 章）来实现的，这些方法旨在改进和强化类推理行为。

## 1.6 为什么从零开始构建推理模型？

随着 DeepSeek-R1 在 2025 年 1 月的发布，提升 LLM 的推理能力已成为 AI 中最热门的话题之一，这是有充分理由的。更强的推理能力使 LLM 能够处理更复杂的问题，使其在用户关心的各种任务中更加强大。

这一转变也反映在 OpenAI CEO 于 2025 年 2 月 12 日的声明中：

> "我们将接下来发布 GPT-4.5，我们内部称为 Orion 的模型，作为我们最后一个非思维链模型。之后，我们的一个首要目标是通过创建能够使用我们所有工具、知道何时应该长时间思考或不思考、并且通常对非常广泛的任务都有用的系统，来统一 o 系列模型和 GPT 系列模型。"

上述引文强调了领先的 LLM 提供商向推理模型的重大转变，其中"思维链"指的是一种引导语言模型逐步推理以提升其推理能力的提示技术，我们将在第 4 章和第 5 章中更详细地介绍。

同样值得注意的是提到知道"何时应该长时间思考或不思考"。这暗示了一个重要的设计考虑：推理并不总是必要或可取的。例如，推理模型被设计为擅长解决谜题、高级数学问题和具有挑战性的编码任务等复杂任务。它们在智能体系统中也很有用，特别是在任务分解、规划和选择使用哪些工具方面。对于更简单的任务，如摘要、翻译或基于知识的问答，它们不是必需的。事实上，对所有事情都使用推理模型可能是低效且昂贵的。例如，推理模型通常更昂贵、更冗长，有时由于"过度思考"而更容易出错。此外，这里适用一个简单的规则：为任务使用正确的工具（或类型的 LLM）。

推理模型通常比非推理模型更昂贵，原因有二。首先，它们倾向于产生更长的输出，因为它们包含解释答案如何得出的中间步骤。如图 1.8 所示，LLM 一次生成一个 token，每个 token 都需要一次完整的前向传播。如果推理模型的答案是两倍长，生成大约涉及两倍的前向传播，这增加了计算成本。¹⁶

**图 1.8** LLM 中的逐 token 生成。在每一步，LLM 获取到目前为止生成的完整序列并预测下一个 token，根据分词器的不同，下一个 token 可能代表一个词、子词或标点符号。新生成的 token 被附加到序列中，并作为下一步的输入。这种迭代解码过程既用于标准语言模型，也用于聚焦推理的模型。

第二，许多推理工作流需要为单个任务多次运行模型，例如采样多个候选解决方案、调用工具或运行验证器。这些额外的调用增加了处理的总 token 数，并进一步增加了超出图 1.8 所示的单次调用行为的成本。

这正是为什么从零开始实现这些模型和方法是有帮助的。这是理解它们如何工作的最佳方式之一。如果我们理解 LLM 和这些推理模型如何工作，我们就能更好地理解这些权衡。

## 1.8 总结

- 传统 LLM 训练分为几个阶段：
  - **预训练**：模型从海量文本中学习语言模式。
  - **指令微调**：改进模型对用户提示的回应。
  - **偏好微调**：将模型输出与人类偏好对齐。
- 推理方法应用于传统 LLM 之上。
- LLM 中的**推理**指的是改进模型，使其在产生最终答案之前显式生成中间步骤（思维链），这通常会提高多步任务上的准确性。
- LLM 中的推理不同于基于规则的推理，也可能与人类推理的工作方式不同；目前，普遍共识是 LLM 中的推理依赖于统计模式匹配。
- LLM 中的**模式匹配**纯粹依赖于从数据中学到的统计关联，这实现了流畅的文本生成，但缺乏显式逻辑推断。
- 改进 LLM 中的推理可以通过以下方式实现：
  - **推理时计算扩展**：在不重新训练的情况下增强推理（例如，思维链提示）。
  - **强化学习**：用奖励信号显式训练模型。
  - **监督微调和蒸馏**：使用来自更强大推理模型的示例。
- 从零开始构建推理模型可以提供关于 LLM 能力、局限性和计算权衡的实用见解。
